## What Ohara Is

Ohara is a local-first, agent-optimized documentation platform. It creates a **documentation hub** — a git repo that stores Diataxis-structured docs for all the services you work on. AI agents read from it, write to it, and keep it current.

**Core idea:** Your workspace has code repos and a docs hub side by side. Agents use the hub as their knowledge layer. The docs compound — agents get smarter as docs improve, agents contribute back, docs improve further.

```
~/work/                          ← Your workspace
├── CLAUDE.md                    ← Agent discovery
├── .claude/
│   ├── agents/                  ← 4 specialized subagents
│   ├── skills/                  ← 5 lightweight skills
│   └── settings.json            ← MCP server config
├── hzn-prices-service/          ← Code repo
├── hzn-auth-service/            ← Code repo
└── ohara-docs/                  ← Documentation hub (git repo)
    ├── .ohara.yaml              ← Hub config (tracked services)
    ├── llms.txt                 ← Doc index for LLMs
    ├── llms-full.txt            ← Full content (completed docs only)
    ├── AGENTS.md                ← Agent instructions
    ├── hzn-prices-service/
    │   ├── CHANGELOG.md         ← Auto-generated from PR/commit history
    │   ├── tutorials/
    │   ├── guides/
    │   ├── reference/
    │   ├── explanation/
    │   └── .ohara-prompts/      ← LLM prompts for doc generation
    └── hzn-auth-service/
        └── ...
```

## What We Built

### 1. CLI Tool (Go)

Distributed via Homebrew: `brew tap Lackech/ohara && brew install ohara`

| Command | What it does |
|---------|-------------|
| `ohara init` | Creates docs hub + CLAUDE.md + agents + skills + MCP config |
| `ohara add <repo>` | Tracks a code repo, creates Diataxis directories |
| `ohara generate <service>` | Analyzes code, scaffolds docs + LLM prompts |
| `ohara build` | Generates llms.txt, llms-full.txt, AGENTS.md, CHANGELOGs |
| `ohara validate` | Checks structure, coverage, stubs, quality |
| `ohara serve` | Local MCP server (stdio) with 7 tools |
| `ohara login` | Store API key for cloud features |
| `ohara status` | Show project sync status |
| `ohara search` | Search docs via API |

### 2. AI Agent Infrastructure

#### 4 Subagents (`.claude/agents/`)

| Agent | Model | Memory | Purpose |
|-------|-------|--------|---------|
| **ohara-writer** | Sonnet | Yes | Reads code, writes Diataxis docs. MCP tools for write-back. |
| **ohara-reviewer** | Haiku | Yes | Reviews docs against code for accuracy. Finds stale content. |
| **ohara-researcher** | Haiku | Yes | Searches docs to answer questions. Auto-invoked. |
| **ohara-watcher** | Haiku | Yes | Detects code changes that make docs stale. Auto-invoked after git pull. |

All agents have **persistent memory** (`memory: project`) — they remember service profiles, patterns, gotchas, and get better over time. Memory is stored in `.claude/agent-memory/` and shared via git.

#### 5 Skills (`.claude/skills/`)

| Skill | Invocation | Purpose |
|-------|-----------|---------|
| `/validate-docs` | Auto | Check structure and coverage |
| `/check-staleness [service]` | Auto (after git pull) | Compare code changes vs docs |
| `/post-merge` | Auto (after PR merge) | Check if merged code needs doc updates |
| `/create-docs-pr <desc>` | Manual | Branch → commit → push → PR |
| `/docs-changelog [service]` | Auto | Recent changes from git log |

#### MCP Server (`ohara serve`)

7 tools available to any MCP client:

| Tool | Purpose |
|------|---------|
| `search_docs` | Grep-based search across all docs |
| `list_docs` | List docs by service and Diataxis type |
| `read_doc` | Read a specific document |
| `write_doc` | Create or update a doc (rebuilds llms.txt) |
| `validate` | Run ohara validate |
| `create_pr` | Branch, commit, push, open PR |
| `changelog` | Git log history |

#### Agent Artifacts (auto-generated by `ohara build`)

| File | Purpose |
|------|---------|
| `llms.txt` | Index of all docs by service + type. Stubs marked as incomplete. |
| `llms-full.txt` | Full inline content. Skips TODO stubs. |
| `AGENTS.md` | Instructions for agents: Diataxis types, PR workflow, changelog. |
| `CHANGELOG.md` (per service) | Auto-generated from code repo's merged PRs + git commits. |
| `CLAUDE.md` | Discovery file: agents, skills, MCP tools, Diataxis reference. |

### 3. Cloud Platform (API + Web)

Built but secondary to the local-first CLI experience.

**Stack:** Elysia (Bun) + Next.js 15 + Fumadocs + PostgreSQL/pgvector (Neon) + BullMQ (Upstash Redis)

| Component | What it does |
|-----------|-------------|
| **Elysia API** | 20+ endpoints: projects, documents, search, webhooks, MCP, scaffolding |
| **Next.js Frontend** | Fumadocs docs viewer, Cmd+K search, dashboard, onboarding wizard |
| **GitHub Sync** | Webhook → BullMQ → clone → parse → chunk → upsert |
| **Hybrid Search** | tsvector + pgvector + Reciprocal Rank Fusion |
| **Cloud MCP Server** | Same 7 tools as local, plus analyze_project, create_document |
| **Embeddings** | OpenAI text-embedding-3-small for semantic search |
| **Auth** | Better Auth with GitHub OAuth + API keys |
| **llms.txt endpoint** | Diataxis-typed at `/:project/llms.txt` |

## The Documentation Lifecycle

```
SETUP (once)
  ohara init → ohara add → ohara generate
       ↓
GENERATE
  "Use ohara-writer to document <service>"
  → writer reads code + prompts → writes docs → MCP write_doc
  → writer saves service profile to memory
       ↓
BUILD
  ohara build → llms.txt + llms-full.txt + AGENTS.md + CHANGELOG.md
       ↓
PUBLISH
  /create-docs-pr → branch → commit → push → PR → merge
       ↓
MAINTAIN (ongoing, automatic)
  ┌────────────────────────────────────────────────┐
  │  git pull           → /check-staleness          │
  │  PR merged          → /post-merge               │
  │  working on feature → ohara-watcher auto-checks │
  │  "are docs current" → ohara-reviewer            │
  │                           ↓                     │
  │  stale detected → ohara-writer updates          │
  │                 → /create-docs-pr               │
  │                 → cycle repeats                 │
  └────────────────────────────────────────────────┘
       ↓
CONSUME (always available)
  "how does auth work?" → ohara-researcher
  llms.txt              → any LLM
  MCP tools             → any MCP client
  CHANGELOG.md          → understand service evolution
```

## Design Principles

### 1. Local-First
The hub is a git repo on disk. Everything works offline. Cloud is optional.

### 2. Diataxis Framework
Every doc is typed: Tutorial, Guide, Reference, or Explanation. Agents get the right content for the right task.

### 3. One Core, Many Interfaces
Same capability via CLI, skills, subagents, MCP tools, REST API, and web UI. Developers choose their interface.

### 4. Agents Are First-Class
Subagents with persistent memory, MCP tools, and write-back. They don't just read docs — they create, update, and maintain them.

### 5. The Flywheel Compounds
Agents read docs → work better → contribute back → docs improve → agents get smarter. Persistent memory accelerates this.

## Agent Patterns (from Anthropic + Google Cloud research)

| Pattern | Where we use it |
|---------|----------------|
| **Prompt Chaining** | Scaffolding: analyze → plan → generate → validate |
| **Evaluator-Optimizer** | Watcher: detect changes → check docs → update → validate |
| **Orchestrator-Workers** | Agent teams: one writer per service, parallel generation |
| **Augmented LLM** | MCP server: LLM + tools + retrieval |

### Key principle from Anthropic
> "Start with simple prompts. Only add complexity when simpler solutions demonstrably fall short."

Most Ohara features are workflows (predefined code paths), not autonomous agents. Skills handle quick operations inline. Subagents handle heavy work in isolation. Agent teams handle parallelism.
